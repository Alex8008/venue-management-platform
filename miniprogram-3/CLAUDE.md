## 1. 代码交付规范

### 1.1 前端代码交付规则
- **WXML / JS / WXSS / JSON 文件**：必须提供完整的修改后的代码（未修改的部分不用给出）
- 对于修改的部分，必须给出完整、可直接使用的代码片段
- 必须明确标注修改位置（如函数名、行号范围、或上下文标识）
- 不得遗漏任何修改点

### 1.2 其他文件修改规则
- 对于配置文件、工具类等文件：
  - 明确说明修改位置
  - 提供修改前后的代码对比
  - 给出完整的修改代码片段

### 1.3 代码整合规则
- 如果同一个文件需要多次修改：
  - 不要分多次给出修改方案
  - 必须整合所有修改，给出最终完整的修改方案
  - 明确标注所有修改点

### 1.4 代码分批规则
- 如果代码量过大：
  - 可以分批次给出代码
  - 但每批次必须是完整的文件或完整的模块
  - 明确说明分批顺序和总批次数

## 2. 代码安全规范

### 2.1 核心原则
**禁止破坏原则**：让改什么就改什么，不要动无关的地方！

### 2.2 具体要求
1. **不要破坏已有的无关代码**
   - 修改时只改需要修改的部分
   - 不要"顺手"优化或重构无关代码
   - 不要改变原有代码风格（除非明确要求）

2. **不要删除无关的代码**
   - 即使某些代码看起来没用，也不要删除
   - 用户可能在其他地方使用了这些代码
   - 只删除明确要求删除的代码

3. **不要遗漏原有的未修改代码**
   - 修改时必须确保未修改部分保持原样
   - 如果使用 Edit 工具修改代码片段，必须保证上下文完整
   - 修改后整个文件应保持正常运行，不能因为修改导致其他部分功能失效

4. **严格遵循已有的命名规范**
   - 新增变量名、函数名必须遵循项目已有的命名风格
   - API URL 路径必须遵循项目已有的路径命名规则（如 `/api/xxx` 格式）
   - data 数据字段名必须与后端返回的字段名一致
   - 认证方式必须使用项目已有的方式（OpenId/User-Id header），不要自行发明新方式

5. **严格注意代码缩进**
   - 保持与原文件一致的缩进风格（空格或tab）
   - 新增代码的缩进要与周围代码对齐
   - 不要因为缩进问题导致代码报错

6. **仔细阅读未理解的现有代码（如需）**
   - 理解现有代码的逻辑和结构
   - 在现有代码基础上进行修改
   - 不要凭想象或假设修改代码

7. **保持功能完整性**
   - 不要让其他已有功能失效
   - 修改后要考虑对其他功能的影响
   - 如果可能影响其他功能，必须提前说明

## 3. 操作权限规范

### 3.1 严格禁止的操作
在没有用户明确许可的情况下，**严格禁止**以下操作：

1. **代码修改**：
   - ❌ 禁止未经许可修改任何代码
   - ❌ 禁止删除任何文件或代码
   - ❌ 禁止重命名文件或目录

2. **系统操作**：
   - ❌ 禁止修改系统配置文件
   - ❌ 禁止执行 `rm`、`sudo`、`format` 等危险命令
   - ❌ 禁止修改 git 配置
   - ❌ 禁止执行 `git push --force`
   - ❌ 禁止执行 `git reset --hard`

3. **依赖管理**：
   - ❌ 禁止安装未经许可的 npm 包
   - ❌ 禁止升级或降级依赖版本
   - ❌ 禁止修改 package.json

4. **其他操作**：
   - ❌ 禁止执行任何可能影响系统的命令
   - ❌ 禁止修改环境变量
   - ❌ 禁止访问或修改用户个人文件

### 3.2 所有操作需要用户同意
- 任何代码修改都必须先得到用户明确同意
- 任何文件创建都必须先征求用户意见
- 任何命令执行都必须先说明目的和影响

## 4. 工作流程规范

### 4.1 需求澄清阶段
当收到用户需求时：

1. **检查需求是否明确**
   - 如果需求模糊或有歧义，必须先提问澄清
   - 不要自作主张猜测用户意图
   - 列出所有不明确的点，逐一确认

2. **提出澄清问题示例**：
   - "您希望这个功能在哪个页面实现？"
   - "这个数据应该从哪个接口获取？"
   - "遇到错误时应该如何提示用户？"
   - "是否需要数据持久化存储？"

### 4.2 方案设计阶段
在开始修改代码之前，必须提供完整的方案说明：

1. **实现思路**
   - 详细说明如何实现该功能
   - 解释技术选型和实现逻辑
   - 说明为什么这样实现

2. **文件清单**
   - 列出所有需要修改的文件
   - 说明每个文件需要做什么修改
   - 是否需要新建文件

3. **影响分析**
   - 说明此次修改可能影响哪些现有功能
   - 是否需要修改相关的其他文件
   - 是否会影响用户体验

4. **风险说明**
   - 说明可能存在的风险点
   - 如何规避这些风险
   - 如果出现问题如何回滚

5. **替代方案**
   - 如果有多种实现方式，列出所有方案
   - 对比各方案的优缺点
   - 推荐最优方案并说明理由

### 4.3 等待批准
- 提供完整方案后，必须等待用户明确批准
- 不要在用户批准前就开始修改代码
- 如果用户提出修改意见，调整方案后再次确认

### 4.4 代码实现阶段
得到用户批准后：

1. **按照批准的方案执行**
2. **严格遵守代码交付规范和代码安全规范**
3. **实现过程中如发现问题，立即暂停并向用户说明**

### 4.5 交付说明阶段
代码完成后，提供清晰的使用说明：

1. **修改总结**
   - 列出所有修改的文件
   - 简要说明每个文件的修改内容

2. **使用指南**
   - 告诉用户如何应用这些修改
   - 如果是完整文件，说明直接替换
   - 如果是代码片段，说明在哪里修改

3. **测试建议**
   - 建议用户测试哪些功能
   - 重点关注哪些可能受影响的地方

### 5.3 态度要求
- 保持谦逊和谨慎的态度
- 不确定的事情不要假装确定
- 遇到困难要坦诚说明
- 尊重用户的所有决定

## 6. 代码质量规范

### 6.1 代码完整性
- 提供的修改部分代码必须是完整且可直接使用的
- 修改部分不要有 `// ... 其他代码` 这样的省略
- 未修改部分可以省略不给出，但必须确保不遗漏任何修改点
- 如果修改涉及多个位置，必须逐一给出每个修改点的完整代码

### 6.2 代码正确性
- 修改后的代码必须语法正确
- 必须考虑边界情况和异常处理
- 必须保证数据类型正确
- 必须保证API调用正确

### 6.3 代码一致性
- 保持与项目现有代码风格一致
- 变量命名遵循项目规范
- 缩进和格式遵循项目规范
- 注释风格遵循项目规范

---

## 7. 特殊说明

### 7.1 关于节省 Token
- 只给出修改后的完整代码，未修改的部分不需要重复输出
- 回复应简洁明了，避免不必要的冗长解释
- 多个修改点可以合并说明，但每个修改点的代码必须完整
- 节省 Token 的前提是：不遗漏任何修改、不损坏任何已有功能

### 7.2 关于验证
- 修改代码前，必须先读取文件查看现有代码
- 不要基于假设或记忆修改代码
- 不要基于过时的信息修改代码

### 7.3 关于后端配合
- 如果前端修改涉及到后端接口变更
- 必须同时说明后端需要做的相应修改
- 提供前后端接口文档

2. 用户有权要求重新提供代码
3. Claude 必须立即纠正错误

## 10. 总结

本规范的核心原则是：
1. **安全第一**：不破坏、不删除、不乱动
2. **精准交付**：只给出修改部分的完整代码，未修改部分不重复输出，节省 Token
3. **不遗漏**：修改时不遗漏任何修改点，不损坏未修改部分的功能
4. **遵循规范**：严格遵循项目已有的变量名、URL、认证方式等命名规范
5. **充分沟通**：先问清楚、再说方案、后写代码
6. **质量保证**：仔细阅读、认真修改、严格测试

请 Claude 严格遵守本规范，确保每一次代码修改都是安全、可靠、高质量的。
