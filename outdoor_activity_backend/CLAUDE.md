## 1. 代码交付规范

### 1.1 后端代码交付规则
- **Python 文件修改过的函数**：必须提供完整的修改后的函数代码（从 `def` 到函数结束），未修改的函数不用给出
- **新增的函数/类**：必须提供完整代码
- **路由函数**：必须提供完整的修改后的路由函数代码，未修改的路由函数不用给出
- **配置文件**：只需提供修改部分的完整代码，并明确标注修改位置

### 1.2 其他文件修改规则
- 对于配置文件、工具类、中间件等文件：
  - 明确说明修改位置
  - 提供修改前后的代码对比
  - 给出完整的修改代码片段

### 1.3 数据库相关修改
- **模型（Model）修改**：提供完整的模型类代码
- **数据库迁移**：提供完整的迁移脚本
- **SQL 语句**：提供完整的 SQL 语句

### 1.4 代码整合规则
- 如果同一个文件需要多次修改：
  - 不要分多次给出修改方案
  - 必须整合所有修改，不要遗漏任何修改，给出最终完整的修改方案
  - 明确标注所有修改点

### 1.5 代码分批规则
- 如果代码量过大：
  - 可以分批次给出代码
  - 但每批次必须是完整的函数或完整的模块
  - 明确说明分批顺序和总批次数

## 2. 代码安全规范

### 2.1 核心原则
**禁止破坏原则**：让改什么就改什么，不要动无关的地方！

### 2.2 具体要求
1. **不要破坏已有的无关代码**
   - 修改时只改需要修改的部分
   - 不要"顺手"优化或重构无关代码
   - 不要改变原有代码风格（除非明确要求）

2. **不要删除无关的代码**
   - 即使某些代码看起来没用，也不要删除
   - 可能在其他地方或其他接口中使用
   - 只删除明确要求删除的代码

3. **不要遗漏原有的未修改代码**
   - 修改时必须确保未修改部分保持原样
   - 如果使用 Edit 工具修改代码片段，必须保证上下文完整
   - 修改后整个文件应保持正常运行，不能因为修改导致其他接口/功能失效

4. **严格遵循已有的命名规范**
   - 变量名、函数名使用 snake_case，与项目已有风格一致
   - API URL 路径必须遵循项目已有的路径命名规则（如 `/api/xxx` 格式）
   - 数据库字段名必须与已有表结构的命名风格一致
   - 认证方式必须使用项目已有的方式（OpenId/User-Id header + @login_required），不要自行发明新方式
   - 返回格式必须遵循已有的 `{"code": 200, "message": "...", "data": ...}` 格式

5. **严格注意代码缩进**
   - Python 对缩进极其敏感
   - 保持与原文件一致的缩进风格（空格数量）
   - 新增代码的缩进要与周围代码对齐
   - 不要混用空格和 tab

6. **仔细阅读未了解的现有代码（如需）**
   - 理解现有代码的逻辑和结构
   - 理解现有的数据库模型和关系
   - 理解现有的 API 接口规范
   - 在现有代码基础上进行修改
   - 不要凭想象或假设修改代码

7. **保持功能完整性**
   - 不要让其他已有 API 接口失效
   - 修改后要考虑对其他接口的影响
   - 如果可能影响其他功能，必须提前说明

8. **保持数据库完整性**
   - 不要破坏现有的数据库表结构
   - 不要删除正在使用的字段
   - 修改字段前要考虑现有数据的兼容性

## 3. 操作权限规范

### 3.1 严格禁止的操作
在没有用户明确许可的情况下，**严格禁止**以下操作：

1. **代码修改**：
   - ❌ 禁止未经许可修改任何代码
   - ❌ 禁止删除任何文件或代码
   - ❌ 禁止重命名文件或目录

2. **系统操作**：
   - ❌ 禁止修改系统配置文件
   - ❌ 禁止执行 `rm`、`sudo`、`format` 等危险命令
   - ❌ 禁止修改 git 配置
   - ❌ 禁止执行 `git push --force`
   - ❌ 禁止执行 `git reset --hard`

3. **依赖管理**：
   - ❌ 禁止安装未经许可的 Python 包
   - ❌ 禁止升级或降级依赖版本
   - ❌ 禁止修改 requirements.txt

4. **数据库操作**：
   - ❌ 禁止执行未经许可的数据库迁移
   - ❌ 禁止直接删除数据库表
   - ❌ 禁止直接修改生产数据库数据
   - ❌ 禁止执行 DROP、TRUNCATE 等危险 SQL 命令

5. **服务器操作**：
   - ❌ 禁止重启服务器

   - ❌ 禁止修改环境变量

6. **其他操作**：
   - ❌ 禁止执行任何可能影响系统的命令
   - ❌ 禁止访问或修改用户个人文件
   - ❌ 禁止修改日志配置

### 3.2 所有操作需要用户同意
- 任何代码修改都必须先得到用户明确同意
- 任何文件创建都必须先征求用户意见
- 任何命令执行都必须先说明目的和影响
- 任何数据库修改都必须先说明影响范围

## 4. 工作流程规范

### 4.1 需求澄清阶段
当收到用户需求时：

1. **检查需求是否明确**
   - 如果需求模糊或有歧义，必须先提问澄清
   - 不要自作主张猜测用户意图
   - 列出所有不明确的点，逐一确认

2. **提出澄清问题示例**：
   - "这个接口需要哪些参数？参数类型是什么？"
   - "返回数据的格式应该是怎样的？"
   - "是否需要身份验证？需要什么权限？"
   - "遇到错误时应该返回什么状态码和错误信息？"
   - "是否需要数据持久化？存储到哪个表？"
   - "是否需要事务处理？"

### 4.2 方案设计阶段
在开始修改代码之前，必须提供完整的方案说明：

1. **实现思路**
   - 详细说明如何实现该功能
   - 解释技术选型和实现逻辑
   - 说明为什么这样实现
   - 如果涉及数据库，说明数据表设计

2. **文件清单**
   - 列出所有需要修改的文件
   - 说明每个文件需要做什么修改
   - 是否需要新建文件
   - 是否需要新建数据库表或字段

3. **接口设计**
   - 如果是新增接口，提供完整的接口文档：
     - 接口路径
     - 请求方法（GET/POST/PUT/DELETE）
     - 请求参数（参数名、类型、是否必填、说明）
     - 返回数据格式（成功和失败的情况）
     - 状态码说明

4. **影响分析**
   - 说明此次修改可能影响哪些现有 API 接口
   - 是否需要修改相关的其他文件
   - 是否会影响前端调用
   - 是否会影响数据库性能

5. **风险说明**
   - 说明可能存在的风险点
   - 如何规避这些风险
   - 如果出现问题如何回滚
   - 是否需要数据备份

6. **替代方案**
   - 如果有多种实现方式，列出所有方案
   - 对比各方案的优缺点
   - 推荐最优方案并说明理由

### 4.3 等待批准
- 提供完整方案后，必须等待用户明确批准
- 不要在用户批准前就开始修改代码
- 如果用户提出修改意见，调整方案后再次确认

### 4.4 代码实现阶段
得到用户批准后：

1. **按照批准的方案执行**
2. **严格遵守代码交付规范和代码安全规范**
3. **实现过程中如发现问题，立即暂停并向用户说明**

### 4.5 交付说明阶段
代码完成后，提供清晰的使用说明：

1. **修改总结**
   - 列出所有修改的文件
   - 简要说明每个文件的修改内容
   - 如果有数据库变更，说明变更内容

2. **使用指南**
   - 告诉用户如何应用这些修改
   - 如果是完整函数，说明替换哪个函数
   - 如果是代码片段，说明在哪里修改
   - 如果需要数据库迁移，提供迁移命令

3. **测试建议**
   - 建议用户测试哪些接口
   - 重点关注哪些可能受影响的地方
   - 提供测试用的请求示例

4. **前端配合**
   - 如果接口有变化，说明前端需要做的相应修改
   - 提供前端调用示例

## 6. 代码质量规范

### 6.1 代码完整性
- 提供的修改部分代码必须是完整且可直接使用的
- 修改部分不要有 `# ... 其他代码` 这样的省略
- 未修改的函数/路由可以省略不给出，但必须确保不遗漏任何修改点
- 修改过的函数必须给出完整的函数代码，不能只给出部分行

### 6.2 代码正确性
- 修改后的代码必须语法正确
- 必须考虑边界情况和异常处理
- 必须保证数据类型正确
- 必须保证 SQL 语句正确
- 必须保证 API 接口规范正确

### 6.3 代码一致性
- 保持与项目现有代码风格一致
- 变量命名遵循项目规范（Python 使用 snake_case）
- 缩进和格式遵循 PEP 8 规范
- 注释风格遵循项目规范

### 6.4 安全性要求
- 必须对用户输入进行验证和过滤

- 密码等敏感信息必须加密存储

## 7. 特殊说明

### 7.1 关于节省 Token
- 只给出修改后的完整函数/路由代码，未修改的函数/路由不需要重复输出
- 回复应简洁明了，避免不必要的冗长解释
- 多个修改点可以合并说明，但每个修改点的代码必须完整
- 节省 Token 的前提是：不遗漏任何修改、不损坏任何已有功能

### 7.2 关于验证
- 修改代码前，必须先读取文件查看现有代码
- 不要基于假设或记忆修改代码
- 不要基于过时的信息修改代码
- 如果不确定数据库结构，要先查看模型文件

### 7.3 关于前端配合
- 如果后端接口变更涉及到前端调用
- 必须同时说明前端需要做的相应修改
- 提供完整的接口文档和调用示例

### 7.4 关于数据库迁移
- 任何数据库结构变更都需要提供迁移方案
- 说明如何迁移现有数据
- 说明如何回滚（如果需要）

## 10. 接口文档规范

### 10.1 接口文档模板
当新增或修改接口时，必须提供以下格式的文档：

```markdown
#### 接口名称
简短描述接口用途

**接口路径**：`/api/xxx/xxx`
**请求方法**：`GET` / `POST` / `PUT` / `DELETE`
**是否需要认证**：是 / 否

**请求参数**：
| 参数名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| user_id | int | 是 | 用户ID | 123 |
| page | int | 否 | 页码，默认1 | 1 |

**返回数据**（成功）：
​```json
{
  "code": 200,
  "message": "成功",
  "data": {
    // 具体数据
  }
}
​```

**返回数据**（失败）：
​```json
{
  "code": 400,
  "message": "错误信息",
  "data": null
}
​```

**状态码说明**：
- 200：成功
- 400：参数错误
- 401：未授权
- 404：资源不存在
- 500：服务器错误
```

## 11. 数据库规范

### 11.1 模型设计规范
- 使用 SQLAlchemy ORM
- 表名使用小写 + 下划线
- 字段名使用小写 + 下划线
- 必须有主键
- 时间字段建议使用 `created_at` 和 `updated_at`

### 11.2 查询规范
- 优先使用 ORM 查询
- 复杂查询可以使用原生 SQL，但必须使用参数化查询
- 注意查询性能，避免 N+1 查询
- 适当使用索引

### 11.3 事务处理
- 涉及多个表操作时，必须使用事务
- 确保事务的 ACID 特性
- 异常时要正确回滚

## 12. 总结

本规范的核心原则是：
1. **安全第一**：不破坏、不删除、不乱动
2. **精准交付**：只给出修改部分的完整代码，未修改部分不重复输出，节省 Token
3. **不遗漏**：修改时不遗漏任何修改点，不损坏未修改部分的功能
4. **遵循规范**：严格遵循项目已有的变量名、URL、认证方式、返回格式等规范
5. **充分沟通**：先问清楚、再说方案、后写代码
6. **质量保证**：仔细阅读、认真修改、严格测试
7. **接口规范**：清晰的接口文档、规范的返回格式
8. **数据安全**：保护数据库、验证输入、防止注入

请 Claude 严格遵守本规范，确保每一次代码修改都是安全、可靠、高质量的。
